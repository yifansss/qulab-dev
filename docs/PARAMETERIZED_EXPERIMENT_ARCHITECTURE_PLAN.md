# Parameterized Experiment and Sequence Bundle Architecture Plan

Status: P9 Phase A/B/C/D implemented; Phase E/F planned.

This document replaces the earlier pulse-entity abstraction plan. The updated
direction is to keep Qulab focused on experiment orchestration, parameter
scans, hardware preflight, metadata, and GUI presentation. ASG sequence details
should be authored by the standalone sequence editor or generated by a
dedicated sequence generator. Qulab should then load and scan a concrete
sequence bundle instead of trying to understand and rewrite every pulse field
inside the main framework.

## 1. Decision Summary

There are two possible designs for scanning ASG sequence-dependent parameters.

### Option A: Pulse Entity Abstraction in Qulab

Qulab would parse a sequence file into pulse entities, such as `mw_gate` and
`ni_trigger`, then bind experiment parameters directly to fields like
`duration_s` or `start_s`.

Advantages:

- The GUI could show pulse fields as ordinary parameter targets.
- A user could edit `tau_s` directly in Operator Parameters without thinking
  about generated files.
- In principle, preflight could reason about every pulse.

Disadvantages:

- Qulab would become a partial pulse compiler and sequence editor.
- The framework would need to understand vendor/editor-specific sequence file
  details.
- Every change in the standalone editor format could break Qulab's parser.
- Timing-sensitive pulse semantics would be split across the editor and Qulab.
- Hardware bring-up would be slowed by a large abstraction layer that is not
  required for running experiments.

### Option B: Sequence Bundle as a Scannable Resource

The sequence editor or a user-provided generator creates a bundle of concrete
sequence files. Each bundle entry corresponds to one parameter point or one
combination of parameter values. Qulab scans experiment parameters, resolves the
matching bundle entry, loads that concrete sequence file, and records the
source/hash/provenance.

Advantages:

- Qulab does not need to understand internal pulse semantics.
- The standalone editor remains the owner of sequence design.
- User code can generate complex sequence families without changing Qulab core.
- Experiment execution remains explicit: scan point -> select sequence -> load
  sequence -> arm/start/read.
- It is easier to test on the bench quickly because every ASG load uses a real
  file that can be opened, inspected, hashed, and copied.
- The same mechanism works for Rabi, Ramsey, Echo, ODMR variants, composite
  pulses, and future vendor formats.

Disadvantages:

- Bundle generation is a separate step before or during a run.
- Preflight can validate bundle coverage and coarse metadata, but it cannot
  fully prove pulse-level correctness unless the bundle provides metadata.
- The GUI edits scan parameters and bundle selection, not individual pulse
  internals by default.

### Selected Direction

Use Option B.

Qulab should not make ASG pulse internals first-class in the main workflow
schema at this stage. The near-term framework should support:

- reading scan coordinates from the existing workflow point context;
- binding current point coordinates to sequence bundle entries;
- loading the selected sequence file at the correct point in the workflow;
- storing sequence artifacts, hashes, point coordinates, and optional bundle
  metadata.

High-level parameter authoring, fixed/linspace/range/explicit GUI controls, and
automatic workflow scan generation are useful, but they are not required for
hardware functionality. They should be implemented later as a GUI convenience
layer after sequence bundle loading is reliable.

Pulse-level abstraction may still exist inside the standalone sequence editor
or inside optional user generator modules, but it should not be required by the
Qulab executor.

## 2. Motivation

Qulab already supports scans through procedure nodes:

```yaml
procedure:
  - scan:
      name: tau_s
      values: {start: 20e-9, stop: 2e-6, points: 101}
      body: [...]
```

The scanned value enters the executor parameter context and can be referenced
by later steps:

```yaml
- call: mw_drive.set_frequency
  args: {freq_hz: "${mw_freq_hz}"}

- call: asg.load_sequence
  args:
    sequence_file: "${sequence_file}"
```

This is enough for many ordinary instrument settings. The hard part is ASG
sequence variation. For example, a Rabi experiment may vary microwave pulse
duration, a Ramsey experiment may vary free precession delay, and a sync test
may vary trigger delay. Those variations are naturally represented as different
concrete sequence files.

The updated goal is:

```text
operator scan parameters
      |
      v
sequence bundle resolver
      |
      v
concrete sequence file for this point
      |
      v
asg.load_sequence(...)
```

The workflow tree remains the source of execution truth.

## 3. Current Capability and Boundary

Current implemented capability:

- `procedure.scan` can scan named parameters.
- `${param}` references can be used in call args.
- Operator Parameters can expose common fields such as scan start/stop/points,
  average count, call args, storage backend, and ASG `sequence_file`.
- ASG sequence files can be selected, inspected, hashed, copied into run
  artifacts, and opened in the standalone editor.
- Resource-level `resources.asg.sequence_file` is the canonical quick-edit
  sequence file.
- Workflow-level `asg.load_sequence args.sequence_file` can override the
  resource default for a specific load step.

Current P9.1/P9.2 capability:

- `qulab.sequence_bundles` provides immutable-style manifest, entry, selection,
  coverage-report models and stable exception classes.
- Version-1 manifests load fully offline with project-relative manifest paths,
  manifest-relative sequence paths, raw-byte SHA-256 validation, structural
  coordinate validation, and `exact|nearest|id` resolution.
- `ParsedExperiment.sequence_bundles` exposes validated bundle models while
  leaving `config` and `resolved_config` YAML-serializable.
- Coverage validation aggregates all unmatched supplied coordinate points.
- `resource.load_sequence_from_bundle` resolves the current workflow point,
  delegates to the adapter's existing `load_sequence`, and emits
  `SequenceSelected`.
- Prepare derives statically enumerable coverage from outer workflow scans and
  validates coarse trigger, route, and acquisition-window metadata.
- RunStore records manifest artifacts, unique selected sequence artifacts, and
  append-only point selections.

Current limitation after P9.2:

- Qulab does not yet provide a generator hook that creates a sequence bundle
  before a run.
- Operator Parameters can edit some existing fields, but automatic
  fixed/linspace/range scan authoring should remain a later priority.

Boundary:

- Qulab core should not parse every ASG pulse into first-class editable entities.
- Qulab core should not be the general ASG pulse compiler.
- Pulse-level editing belongs in `tools/sequence_editor/` or user generator
  modules.
- Qulab may inspect sequence files for basic metadata, hash, duration estimate,
  channel usage, and trigger hints when available, but execution must not depend
  on complete pulse-level understanding.

## 4. Design Principles

1. Keep workflow scan as the canonical execution semantics.

   The executor should continue to run `scan`, `average`, `measurement`, `run`,
   and `call` nodes. Parameterized UI and schema extensions should generate or
   update these nodes, not bypass them.

2. Keep sequence generation outside the core executor.

   A sequence bundle can be created by the standalone editor, a command-line
   generator, or a Python module. Qulab consumes the generated result.

3. Scan concrete files, not hidden pulse mutations.

   At every measurement point, Qulab should know exactly which sequence file was
   loaded. That file should be copied into run artifacts or referenced by hash.

4. Treat GUI parameter controls as a projection of config, not hidden state.

   Any Operator Parameters entry must map to visible YAML/config fields. A saved
   YAML file must fully represent the experiment plan.

5. Separate slow orchestration from fast pulse timing.

   Python can scan parameters, configure instruments, select a sequence file,
   arm devices, start ASG, and read data. Nanosecond/microsecond timing remains
   inside ASG/NI/AWG hardware and their sequence/trigger configurations.

6. Let optional metadata improve validation without becoming mandatory.

   A bundle manifest may include channel usage, trigger channels, duration,
   labels, and generator provenance. If metadata is missing, Qulab can still run
   with weaker preflight warnings.

## 5. Proposed Concepts

### 5.1 Point Coordinate

For the bundle MVP, a point coordinate is simply a value already present in the
executor parameter context. Usually it comes from an existing workflow
`procedure.scan` node:

```yaml
procedure:
  - scan:
      name: tau_s
      values: {start: 20e-9, stop: 2e-6, points: 101}
      body: [...]
```

At each point, the executor already has:

```text
tau_s = current scan value
```

The sequence bundle resolver should consume this context directly. No new scan
engine is required.

Optional descriptive metadata can be added later:

```yaml
parameters:
  tau_s:
    label: MW pulse width
    unit: s
    default: 100e-9
    role: sequence_coordinate
```

This metadata is for GUI labels, units, and validation only. It should not be a
prerequisite for running a bundle-based experiment.

### 5.2 Later Scan Authoring Layer

Fixed/linspace/range/explicit controls and automatic workflow scan generation
are intentionally deferred. They are a GUI authoring layer, not an execution
requirement.

Later planned schema may still look like:


```yaml
scans:
  tau_s:
    mode: linspace
    start: 20e-9
    stop: 2e-6
    points: 101

  mw_power_dbm:
    mode: explicit
    values: [-30, -25, -20]
```

But the near-term sequence bundle implementation should not depend on this
schema. For now, users or templates can write ordinary workflow `scan` nodes.

Later allowed modes:

- `fixed`: one value, no workflow scan generated.
- `linspace`: `{start, stop, points}`.
- `range`: `{start, stop, step}`.
- `explicit`: `{values: [...]}`.

The GUI may eventually show these modes as segmented controls or a combo box.
Internally, scanned entries should compile to nested workflow `scan` nodes, but
that can wait until after bundle loading, preflight, and storage are stable.

### 5.3 Binding

A binding maps an experiment parameter to a concrete target in config,
instrument resource state, procedure call args, or sequence bundle lookup.

Near-term schema:

```yaml
bindings:
  - parameter: tau_s
    target: sequence_bundle.rabi_tau.coordinate.tau_s
```

Binding categories:

- `sequence_bundle.<bundle_id>.coordinate.<name>`
- `resources.<resource>.<field>`
- `setup.call[<resource.method>].args.<arg>`
- `procedure.call[<resource.method>].args.<arg>`
- `cleanup.call[<resource.method>].args.<arg>`
- Existing operator parameter sources such as
  `procedure.scan[tau_s].values.start`.

Bindings should be explicit and inspectable. If a target cannot be found or
cannot be created safely, preflight should report a clear issue.

For the MVP, the bundle resolver can also work without a full binding block if
the workflow call explicitly provides the coordinate:

```yaml
- call: qulab.resolve_sequence_bundle
  args:
    bundle: rabi_tau
    coordinates:
      tau_s: "${tau_s}"
  save_as: sequence_file
```

This explicit form is the fastest path to a working bench experiment.

### 5.4 Sequence Bundle

A sequence bundle is a collection of concrete sequence files plus an index that
maps coordinates to files.

Recommended directory layout:

```text
configs/sequences/
  rabi_tau_bundle/
    manifest.yaml
    sequences/
      tau_000000020ns.json
      tau_000000040ns.json
      tau_000000060ns.json
```

Planned manifest schema:

```yaml
schema_version: 1
kind: sequence_bundle
id: rabi_tau
resource: asg
format: pycontrol_asg_json

coordinates:
  tau_s:
    unit: s
    values: [20e-9, 40e-9, 60e-9]

entries:
  - id: tau_000000020ns
    coordinates: {tau_s: 20e-9}
    sequence_file: sequences/tau_000000020ns.json
    metadata:
      duration_s: 8e-6
      trigger_channels: [ch6]
      output_channels: [ch1, ch2, ch6]

  - id: tau_000000040ns
    coordinates: {tau_s: 40e-9}
    sequence_file: sequences/tau_000000040ns.json
    metadata:
      duration_s: 8e-6
      trigger_channels: [ch6]
      output_channels: [ch1, ch2, ch6]

generator:
  module: analysis_modules.sequence_generators.rabi_tau
  function: build_bundle
  version: "optional"
  source_hash: "optional"
```

Required fields:

- `kind: sequence_bundle`
- `id`
- `resource`
- `entries[].coordinates`
- `entries[].sequence_file`

Optional but recommended fields:

- `format`
- `coordinates.<name>.unit`
- `coordinates.<name>.values`
- `entries[].metadata.duration_s`
- `entries[].metadata.trigger_channels`
- `entries[].metadata.output_channels`
- `generator`
- per-entry `sha256`, or a run-time computed hash.

### 5.5 Sequence Bundle Resource Config

The ASG resource can keep a default single file for simple operation, or point
to a bundle for scan operation.

```yaml
resources:
  asg:
    adapter: pycontrol_asg
    capabilities: [pulse_sequencer, trigger_source]
    sequence_file: configs/sequences/rabi_default.json
    sequence_bundle: configs/sequences/rabi_tau_bundle/manifest.yaml
    simulation: false
```

Rules:

- `sequence_file` remains valid and is best for single-sequence tests.
- `sequence_bundle` is optional and only used when a workflow step asks for a
  bundle-resolved sequence.
- If both are present, `sequence_file` is the fallback/default, while
  `sequence_bundle` is the indexed scan source.

### 5.6 Bundle Resolver

The bundle resolver maps current point coordinates to a bundle entry.

Resolution inputs:

- bundle manifest path;
- current parameter context, such as `{"tau_s": 40e-9}`;
- optional match tolerance for floating point coordinates;
- optional selection strategy.

Planned selection strategies:

- `exact`: all requested coordinates must match a manifest entry.
- `nearest`: select nearest numeric coordinate within tolerance.
- `id`: use an explicit entry id from the parameter context.

Recommended default:

```yaml
sequence_bundles:
  rabi_tau:
    manifest: configs/sequences/rabi_tau_bundle/manifest.yaml
    resource: asg
    match:
      mode: nearest
      tolerance:
        tau_s: 1e-15
```

The resolver should produce:

```text
resource: asg
bundle_id: rabi_tau
entry_id: tau_000000040ns
coordinates: {tau_s: 40e-9}
sequence_file: configs/sequences/rabi_tau_bundle/sequences/tau_000000040ns.json
sha256: ...
metadata: ...
```

## 6. Workflow Relationship

There is no contradiction between future Operator Parameters scan controls and
workflow tree scan nodes, but this layer should not be part of the near-term
bundle MVP.

The relationship should be:

```text
Operator Parameters panel
  edits high-level parameters and scan specs
        |
        v
Canonical config/workflow tree
  stores explicit scan nodes and bundle load calls
        |
        v
Executor
  runs workflow tree only
```

For the near-term implementation, the user or template writes the canonical
`scan` node directly. At each point, a workflow call resolves and loads the
bundle entry:

```yaml
procedure:
  - scan:
      name: tau_s
      values: {start: 20e-9, stop: 2e-6, points: 101}
      body:
        - measurement:
            name: rabi_point
            body:
              - call: asg.load_sequence_from_bundle
                args:
                  bundle: rabi_tau
                  coordinates:
                    tau_s: "${tau_s}"
              - call: daq.configure_ai_external_trigger
                args:
                  channels: [ai1]
                  sample_rate: 1000000
                  samples: 5000
                  start_trigger: PFI2
              - run:
                  name: pulse_readout
                  steps:
                    - call: daq.arm
                    - call: asg.arm
                    - call: asg.start
                    - call: daq.read
                      save_as: pse_ai1_trace
```

The exact call name can be refined. It may also be implemented as a generic
framework call that resolves a sequence path and then invokes existing
`asg.load_sequence`.

Equivalent explicit form:

```yaml
- call: qulab.resolve_sequence_bundle
  args:
    bundle: rabi_tau
    coordinates: {tau_s: "${tau_s}"}
  save_as: sequence_file

- call: asg.load_sequence
  args:
    sequence_file: "${sequence_file}"
```

The explicit form is easier to test because the selected file is visible as a
normal point variable.

## 7. Sequence Generation

Bundle generation can happen outside the run or as a pre-run step.

### 7.1 Offline Generation

The user opens the standalone sequence editor or runs a script. It writes:

- concrete sequence files;
- `manifest.yaml`;
- optional metadata and hashes.

The experiment config then references the manifest.

This path is implemented and remains the compatibility/fallback mode.

### 7.2 Sequence Family and Prepare-Time Generation

Phase E replaces the earlier loose `build_bundle(config, output_dir)` hook with
a typed provider and framework-owned packager. The authoring config describes a
sequence plan:

```yaml
sequence_plans:
  rabi_main:
    resource: asg
    family:
      provider: analysis_modules.sequence_generators.rabi
      version: "1"
    parameters:
      tau_s: {mode: linspace, start: 20e-9, stop: 2e-6, points: 101}
      laser_init_s: {mode: fixed, value: 3e-6}
      readout_s: {mode: fixed, value: 2e-6}
```

The provider produces one concrete point from one complete parameter mapping:

```python
def generate_point(parameters, *, template=None) -> GeneratedSequencePoint:
    ...
```

Rules:

- providers live under project-local `analysis_modules/sequence_generators/` or
  an explicitly registered built-in provider path;
- providers do not write manifests or own scan loops;
- Qulab enumerates values, assigns entry IDs, writes concrete files and
  manifest, computes hashes, and validates the result through P9.1/P9.2;
- generation never connects hardware and completes before output authorization;
- provider identity/source hash, normalized plan, template hash, diagnostics,
  manifest hash, and point selections are recorded;
- detailed contract and GUI design are in
  `docs/SEQUENCE_FAMILY_GENERATOR_PHASE_E_PLAN.md`.

## 8. Operator Parameters UX

Near-term Operator Parameters should focus on fields that already exist in the
config:

- selected single `sequence_file`;
- selected `sequence_bundle` manifest;
- existing workflow scan values if already discoverable;
- storage backend and common call args;
- bundle coverage/preflight status.

It should not need to generate or rewrite workflow scan nodes for the first
bundle implementation.

Later, after bundle loading is stable, the operator may see a compact table or
form:


```text
Parameter          Mode       Values                         Target
tau_s              linspace   20e-9 -> 2e-6, 101 points      rabi_tau bundle coordinate
mw_freq_hz         fixed      2.87e9                         mw_drive frequency
mw_power_dbm       fixed      -20                            mw_drive power
sequence_bundle    fixed      rabi_tau                       ASG sequence bundle
```

Each later scan-authoring row should expose:

- parameter name and label;
- unit;
- mode: fixed / linspace / range / explicit;
- value fields appropriate to mode;
- target/binding summary;
- warning if target is missing, dangerous, ambiguous, or not currently writable.

For sequence parameters, the binding summary should be readable:

```text
tau_s -> sequence bundle rabi_tau coordinate tau_s
```

The operator should not need to manually edit pulse internals during normal
operation. If the sequence family needs to change, the user opens the sequence
editor or generator settings.

## 9. Builder Workflow UX

Builder mode remains for advanced structure editing:

- Add/remove/reorder scan nodes.
- Add measurement/run/call steps.
- Edit resource and sync details.
- Inspect generated scan hierarchy.
- Inspect or repair parameter bindings.
- Inspect bundle resolver steps.

Builder should not show every individual pulse by default in the main workflow
tree. Sequence details belong in the ASG sequence panel/editor or generator
module. Builder can show summary nodes or links:

```text
asg sequence bundle: rabi_tau
  manifest: configs/sequences/rabi_tau_bundle/manifest.yaml
  coordinate: tau_s
  entries: 101
  selected at current point: tau_000000040ns
```

## 10. Direct Control UX

Direct Control is separate from parameterized scans. It is for bench debugging:

- connect/disconnect resource;
- health/snapshot;
- ASG arm/stop;
- ASG channel always-high/always-low with explicit authorization;
- load a single selected sequence file for scope testing;
- NI AI read-only trace;
- NI AO static voltage with explicit authorization.

Direct Control should not silently modify the planned scan schema. If it changes
a config field, it must be visible and saved deliberately.

## 11. Preflight Requirements

Preflight should validate:

1. Every scanned parameter has valid values.
2. Every binding target exists or is safely creatable.
3. Every referenced bundle manifest exists.
4. Every bundle entry points to an existing sequence file.
5. Bundle coordinate coverage matches the scan plan.
6. Numeric coordinate matching has an explicit tolerance when needed.
7. Selected ASG resource supports `load_sequence`.
8. Sequence file hashes can be computed.
9. Optional trigger metadata is compatible with `sync.triggers`.
10. Optional duration metadata is long enough for DAQ sample/readout windows.
11. Dangerous instrument settings, such as MW output enable or NI AO voltage,
    are not hidden inside normal Operator Parameters without safety metadata.

Warnings are acceptable for missing optional sequence duration or trigger
metadata. Missing manifest, missing sequence file, or missing bundle coverage
should be errors before a hardware run.

## 12. Storage and Provenance

Each run should record:

- parameter definitions;
- scan specs;
- resolved point coordinates;
- binding list;
- bundle manifest source path and hash;
- selected sequence entry id per point;
- selected sequence source file path per point;
- selected sequence artifact copy path per unique entry or per point;
- selected sequence hash;
- optional duration/channel/trigger metadata;
- generator module/function/args/source hash when used;
- adapter snapshots after configuration when safe.

Minimum metadata addition:

```json
{
  "parameter_model": {...},
  "bindings": [...],
  "sequence_bundles": [
    {
      "id": "rabi_tau",
      "resource": "asg",
      "manifest_source": "configs/sequences/rabi_tau_bundle/manifest.yaml",
      "manifest_artifact": "artifacts/sequences/rabi_tau/manifest.yaml",
      "manifest_sha256": "...",
      "entry_count": 101,
      "coordinates": ["tau_s"]
    }
  ],
  "sequence_selection": [
    {
      "point_id": "p000001",
      "bundle": "rabi_tau",
      "entry_id": "tau_000000040ns",
      "coordinates": {"tau_s": 4e-8},
      "source_path": "configs/sequences/rabi_tau_bundle/sequences/tau_000000040ns.json",
      "artifact": "artifacts/sequences/rabi_tau/tau_000000040ns.json",
      "sha256": "..."
    }
  ]
}
```

For large scans, `sequence_selection` can be stored as a table artifact rather
than embedded fully in `metadata.json`.

## 13. Compatibility Plan

Existing configs must continue to work.

Compatibility rules:

- A config with only workflow `scan` nodes remains valid.
- A config with only `resources.asg.sequence_file` remains valid.
- A config with `asg.load_sequence` remains valid.
- A config with `asg.set_sequence_param` remains valid where the adapter
  supports it.
- New `parameters`, `scans`, `bindings`, and `sequence_bundles` are optional.
- The executor should not require the new model to run old experiments.

Migration path:

1. Existing Rabi:

   ```yaml
   scan tau_s
     call asg.set_sequence_param tau_s
   ```

2. Short-term robust Rabi:

   ```yaml
   scan tau_s
     call asg.load_sequence sequence_file=${sequence_file_from_operator}
   ```

3. Bundle-based Rabi:

   ```yaml
   scan tau_s
     resolve rabi_tau bundle at tau_s
     call asg.load_sequence selected_file
   ```

4. Optional generator-based Rabi:

   ```yaml
   pre-run generate rabi_tau bundle
   scan tau_s
     resolve/load generated bundle entry
   ```

## 14. Implementation Phases

### Phase A: Sequence Bundle Manifest Model, Dry-Run Only

Status: completed in P9.1.

Add internal models:

- `SequenceBundle`
- `SequenceBundleEntry`
- `SequenceBundleSelection`
- optional `ParameterBinding` / `BindingTarget` only where needed for bundle
  coordinate documentation.

Add parser support for optional top-level:

- `sequence_bundles`

`bindings` remains a later metadata/authoring extension and was not required by
P9.1.

No hardware behavior changes in this phase.

Acceptance:

- Existing tests pass.
- Bundle manifest paths parse without hardware.
- Missing bundle manifests, invalid entry coordinates, or missing sequence files
  produce clear validation errors.

### Phase B: Sequence Bundle Manifest Loader

Status: completed in P9.1. Coverage is exposed as an explicit offline API;
P9.2 additionally derives coverage for statically enumerable workflow scans.

Add bundle support:

- parse `manifest.yaml`;
- resolve project-relative manifest paths and manifest-relative sequence paths;
- compute manifest and sequence hashes;
- validate coordinate coverage against a supplied workflow scan coordinate list
  when available;
- expose selected entry metadata to preflight and RunStore.

Acceptance:

- A synthetic bundle with three `tau_s` entries resolves the correct sequence
  for each point.
- Missing entry or missing file fails before hardware execution.

### Phase C: Bundle Resolver Workflow Call

Status: completed in P9.2. The canonical action is
`resource.load_sequence_from_bundle`; it delegates to the existing adapter
`load_sequence` and emits `SequenceSelected` only after a successful load.

Add a resolver call or macro:

- resolve current point coordinates to sequence file;
- save selected file path into point context;
- call existing `asg.load_sequence` with that path;
- copy loaded sequence into run artifacts.

Acceptance:

- Existing hand-written workflow `scan tau_s` can drive bundle selection.
- Scan ASG sequence bundle entries without manually changing the sequence file
  per point.
- Metadata records manifest hash, selected entry id, sequence hash, and point
  coordinates.

### Phase D: Hardware Preflight and RunStore Integration

Status: completed in P9.2 for static config/metadata consistency. RunStore keeps
the manifest, unique selected-entry artifacts, append-only point selections,
and run-level summaries. Physical cabling and device route availability remain
bench validation responsibilities.

Use bundle metadata to validate and record:

- bundle coordinate coverage for the planned workflow scan;
- ASG trigger metadata when provided;
- NI sample window covers sequence/readout duration when provided;
- PFI route and trigger mode are compatible with selected bench template;
- manifest and selected sequence artifacts copied into RunStore.

Acceptance:

- Bench05/Rabi preflight catches missing bundle coverage before DAQmx timeout.
- Warnings remain actionable and identify exact bundle entry/resource.
- RunStore can reconstruct which sequence file was loaded for every point.

### Phase E: Sequence Family and Sweep Generator

Status: architecture planned; implementation pending. Detailed contract:
`docs/SEQUENCE_FAMILY_GENERATOR_PHASE_E_PLAN.md`.

Phase E is broader than the earlier optional generator hook. A hook alone still
requires the operator to hand-edit generator source, manifest declarations,
workflow scans, and bundle-load actions. The revised phase adds:

- typed Sequence Family/provider contracts with GUI-readable parameter specs;
- a top-level `sequence_plans` authoring model containing fixed and scanned
  parameter values;
- framework-owned parameter enumeration, manifest writing, hashing, atomic
  materialization, and cache identity;
- a `sequence_sweep` authoring macro compiled into existing workflow `scan` and
  `resource.load_sequence_from_bundle` nodes;
- a generic ASG template sweep provider for selecting pulse start/duration/end
  and defining following/group shift or anchor constraints;
- an integrated GUI `Sequence Sweep` submode for family/template selection,
  ranges, multi-dimensional axes, target/link editing, preview, and Prepare;
- parameter, provider, template, plan, bundle, and point-selection provenance.

The existing executor remains unchanged: Phase E compiles to the P9.1/P9.2
bundle path. Generation and validation finish before hardware connection/output.

Acceptance:

- Changing Rabi `tau_s` start/stop/points requires no Python source edit.
- Normal GUI use requires no handwritten manifest, `sequence_bundles` block, or
  repetitive bundle-load workflow action.
- A generic template can scan one or more pulse properties with deterministic
  dependent-pulse movement and no cumulative point-to-point drift.
- Sequence parameters appear as experiment coordinates and are correctly
  associated with raw data and `SequenceSelected` provenance.
- Generator/provider failure prevents hardware run before any output starts.

### Phase F: Global Scan Authoring, Later Convenience Layer

Status: planned after the runtime bundle path.

Phase E now includes the sequence-specific controls required to author a
generated bundle. Phase F is reserved for a broader experiment-wide scan editor:

- unified fixed/linspace/range/explicit controls for all resource types;
- heterogeneous nested scans across sequence, MW, NI, temperature, magnet, and
  future resources;
- zipped/table/adaptive scan definitions;
- arbitrary loop ordering and binding graph visualization;
- warnings for globally unbound or conflicting experiment parameters.

Acceptance:

- Heterogeneous scan plans can be authored without manually editing workflow
  nesting.
- Saving YAML preserves one canonical authoring model and a reproducible
  compiled workflow.
- Dry-run output coordinates and loop ordering match the global scan plan.

This phase remains lower priority than Phase E. It is not required to make Rabi
or other sequence-family scans convenient in the integrated panel.

## 15. Example Phase E Rabi Authoring Config

```yaml
schema_version: 1
name: rabi_sequence_bundle

resources:
  asg:
    adapter: pycontrol_asg
    capabilities: [pulse_sequencer, trigger_source]
    simulation: false
  daq:
    adapter: pycontrol_ni
    capabilities: [analog_input, trigger_receiver]
    driver: sync
    device: Dev2
    simulation: false

sequence_plans:
  rabi_main:
    resource: asg
    family:
      provider: analysis_modules.sequence_generators.rabi
      version: "1"
    parameters:
      laser_init_s: {mode: fixed, value: 3e-6}
      tau_s:
        mode: linspace
        start: 20e-9
        stop: 2e-6
        points: 101
      readout_s: {mode: fixed, value: 2e-6}
    sampling:
      mode: cartesian
      order: [tau_s]

procedure:
  - sequence_sweep:
      plan: rabi_main
      body:
        - measurement:
            name: rabi_point
            body:
              - call: daq.configure_ai_external_trigger
                args:
                  channels: [ai1]
                  sample_rate: 1000000
                  samples: 5000
                  start_trigger: PFI2
              - run:
                  name: pulse_readout
                  steps:
                    - call: daq.arm
                    - call: asg.arm
                    - call: asg.start
                    - call: daq.read
                      save_as: pse_ai1_trace
```

The GUI edits this authoring form. Prepare materializes the bundle and expands
`sequence_sweep` into the currently implemented canonical `scan` plus
`asg.load_sequence_from_bundle` workflow. The generated form is frozen in
`resolved_config.yaml`; users do not maintain both forms.

## 16. Phase E Decisions

1. Generated bundles use an immutable `.qulab/cache/sequence_bundles/` staging
   cache and are copied into each run's artifacts. They do not modify
   `configs/` by default.
2. The standalone editor edits the base template. The Qulab framework writes
   bundle manifests; individual providers return concrete point artifacts and
   metadata.
3. Point selections remain in append-only `sequence_selections.jsonl` with a
   compact metadata summary. Large point lists are not embedded in
   `metadata.json`.
4. Phase E begins with deterministic Cartesian multi-dimensional sampling and
   explicit loop order. Zip/table modes are follow-ups and must use explicit
   schema rather than implicit guessing.
